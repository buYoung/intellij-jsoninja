이 벤치마크가 뭔가요?

코드를 고치기 직전 개발자가 흔히 던지는 질문 — 예를 들어 "이 함수의 인자를 바꾸면 어디어디를 같이 고쳐야 하지?" — 을 코드 검색 도구 7개에게 똑같이 물어보고, 사람이 소스를 직접 읽어 만든 정답 목록(gold)과 대조해 누가 더 빠짐없이(재현율), 헛다리 없이(정밀도) 찾는지를 잰 실험입니다.

도구마다 "코드를 이해하는 방식"이 다릅니다(그래프 / 심볼 / 텍스트 / 의미 임베딩). 그래서 질문 유형이 바뀌면 잘하던 도구가 갑자기 무너지기도 합니다 — 이 실험의 핵심 결과가 바로 그 "만능 도구는 없다"는 점입니다.

⏱️ 30초 결론

  • 만능 도구는 없다. 4개 질문 평균 재현율은 텍스트 계열(rg · zoekt+ctags 0.93)이 가장 높았지만, 질문마다 1등이 계속 바뀐다.
  • 그래프 도구(codegraph)는 "빠짐없이 전수 조회"엔 최강이지만 "언어 경계"에서 무너진다. M1·M2 는 최상위였다가 Kotlin↔Rust 경계(M3-W 0.50)·enum 참조(M3-L 0.00)에서 급락 → 평균 0.56.
  • 의미 검색(cocoindex)은 "다 찾기"에 약하다. 비슷한 코드는 잘 찾지만 전수 열거가 안 돼 평균 재현율 최저(0.40).
  • 승부의 진짜 축은 "정밀하냐 vs 텍스트냐"가 아니라 "그 도구가 무엇을 색인했느냐"(문자열 / enum 참조 / 언어 간 링크 / 정확한 줄 위치)다.

먼저 알아둘 용어

용어쉽게 말하면
정답(gold)사람이 소스를 직접 읽어 만든 정답 목록. 측정 대상 도구를 하나도 쓰지 않고 만들어 채점이 공정하다.
재현율(recall)찾아야 할 정답 N개 중 실제로 찾아낸 비율. 1.00 = 하나도 안 놓침.
정밀도(precision)도구가 "이게 답"이라고 낸 것 중 진짜 정답인 비율. 1.00 = 헛다리 0.
함정(trap)이름은 비슷하지만 정답이 아닌 코드. 도구가 잘 거르는지(정밀도) 보려고 일부러 섞어 둔 것.
