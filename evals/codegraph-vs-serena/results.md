# 코드 검색 도구 벤치마크 결과 — 도구 7개 × 질문 4개

> **한 줄 요약**: 코드를 고치기 직전 개발자가 던지는 질문 4가지를 코드 검색 도구 7개에 똑같이 물어, 사람이 만든 정답과 대조해 정확도·효율을 쟀다. 결론은 **"만능 도구는 없다"** 이다 — 질문 유형이 바뀌면 1등이 계속 바뀐다.

---

## 1. 이 문서 읽는 법

이 벤치마크는 "코드를 고치기 직전 개발자가 흔히 던지는 질문"(예: *이 함수 인자를 바꾸면 어디어디를 같이 고쳐야 하지?*)을 검색 도구 7개에 동일한 프롬프트로 던지고, 그 답을 **정답(gold)** 과 대조해 채점한 것이다. 각 도구는 같은 sonnet 서브에이전트가 구동하며, **차이는 오직 "쓸 수 있는 도구"뿐**이다.

### 먼저 알아둘 용어

| 용어 | 쉽게 말하면 |
|------|------------|
| **정답(gold)** | 사람이 소스(`git v1.12.1`)를 직접 읽어 만든 정답 목록. 측정 대상 도구를 하나도 쓰지 않고 만들어 채점이 공정하다. |
| **재현율(recall)** | 찾아야 할 정답 N개 중 실제로 찾아낸 비율. **1.00 = 하나도 안 놓침.** |
| **정밀도(precision)** | 도구가 "이게 답"이라고 낸 것 중 진짜 정답인 비율. **1.00 = 헛다리 0.** |
| **함정(trap)** | 이름은 비슷하지만 정답이 아닌 코드. 도구가 잘 거르는지(정밀도) 보려고 일부러 섞어 둔 것. |
| **줄 ±1 보정** | 도구마다 줄 번호 세는 기준이 달라(0-based/1-based, 임베딩은 줄 '범위'를 줌) 정답과 ±1 줄까지는 같은 것으로 인정해 채점했다. |

### 측정한 도구 7개

| 도구 | 한 줄 설명 | 분류 |
|------|-----------|------|
| **A codegraph** | 코드의 호출 관계를 미리 그래프 DB 로 만들어 두고 "누가 이걸 부르나"를 한 번에 질의 | 그래프 |
| **B serena** | IDE 의 "사용처 찾기 / 정의로 이동"과 같은 LSP 기반 심볼 분석(MCP) | 심볼(정밀) |
| **C zoekt** | 구글식 코드 검색 엔진(트라이그램 인덱스). 빠른 텍스트 검색 | 텍스트 |
| **D rg + sg** | `ripgrep`(초고속 grep) + `ast-grep`(구문 구조 검색) | 텍스트 |
| **E bare** | 추가 도구 없이 기본 검색(grep) + 파일 읽기만. 비교 기준선(baseline) | 텍스트 |
| **F cocoindex** | 코드를 임베딩 벡터로 바꿔 "의미가 비슷한" 코드를 찾는 시맨틱 검색 | 의미 |
| **H zoekt + ctags** | zoekt 에 ctags 심볼 정보를 더해 함수·심볼 단위(`sym:`) 검색까지 가능 | 텍스트+심볼 |

### 물어본 질문 4개

| 질문 | 실제 상황 | 정답 수 |
|------|-----------|---------|
| **M1 시그니처 영향** | "이 함수(`JsonFormatterService.formatJson`)의 인자를 바꾸면 같이 고쳐야 할 호출지점이 어디?" | 33 |
| **M2 동작 전파** | "이 함수(`getOrCreate`)의 동작을 바꾸면 결국 어떤 기능들이 영향받나?" (호출 사슬을 거슬러 올라간 전 범위) | 8 |
| **M3-W 언어 간 경계** | "Kotlin 이 Rust(WASM) 함수를 **문자열 이름**으로 부르는 지점과, 그에 대응하는 Rust 정의는?" | 8 |
| **M3-L enum 참조** | "`SupportedLanguage.KOTLIN` 이라는 enum 상수를 실제로 쓰는 곳은 모두 어디?" | 29 |

> M1·M2 는 한 언어(Kotlin) 안의 질문, **M3-W·M3-L 은 Rust 가 섞인 여러 언어(polyglot) 경계** 질문이다. 도구의 진짜 실력은 이 경계에서 갈렸다.

## 2. 30초 결론

1. **만능 도구는 없다.** 4개 질문 평균 재현율은 텍스트 계열(rg · zoekt+ctags **0.93**)이 가장 높았지만, 질문마다 1등이 계속 바뀐다.
2. **그래프 도구(codegraph)는 "빠짐없이 전수 조회"엔 최강이지만 "언어 경계"에서 무너진다.** M1·M2 최상위였다가 Kotlin↔Rust 경계(M3-W **0.50**)·enum 참조(M3-L **0.00**)에서 급락 → 평균 0.56.
3. **의미 검색(cocoindex)은 "다 찾기"에 약하다.** 비슷한 코드는 잘 찾지만 전수 열거가 안 돼 평균 재현율 최저(**0.40**). 단 함정·환각은 0.
4. **승부의 진짜 축은 "정밀하냐 vs 텍스트냐"가 아니라 "그 도구가 무엇을 색인했느냐"** — 문자열 리터럴 / enum 참조 / 언어 간 링크 / 정확한 줄 위치를 색인했는지가 질문마다 우열을 가른다.

## 3. 전체 스코어보드 (재현율 / 정밀도)

각 칸은 **재현율 / 정밀도**. **굵게 = 그 질문에서 두드러진 약점**.

| 질문 (정답 수) | A codegraph | B serena | C zoekt | D rg(+sg) | E bare | F cocoindex | H zoekt+ctags |
|---|---|---|---|---|---|---|---|
| **M1** 시그니처 영향 (33) | 1.00 / **0.94** | **0.76** / 0.96 | 1.00 / 0.97 | 1.00 / 0.97 | 0.97 / 1.00 | **0.67** / 0.81† | 1.00 / 1.00 |
| **M2** 동작 전파 (8) | 0.75 / 1.00 | 0.75 / 1.00 | **0.625** / 1.00 | 0.75 / 1.00 | **0.625** / 1.00 | 0.75 / 1.00 | 0.75 / 1.00 |
| **M3-W** 언어 간 경계 (8) | **0.50** / **0.44** | 1.00 / 1.00 | 1.00 / 1.00 | 1.00 / 1.00 | 1.00 / 1.00 | **0.125** / 0.10 | 1.00 / **0.67** |
| **M3-L** enum 참조 (29) | **0.00** / – | 1.00 / 1.00 | 0.966 / 1.00 | 0.966 / 1.00 | 0.966 / 1.00 | **0.07** / 0.15 | 0.966 / **0.76** |
| **평균 재현율** | **0.56** | 0.88 | 0.90 | **0.93** | 0.89 | **0.40** | **0.93** |
| **평균 정밀도** | 0.79‡ | 0.99 | 0.99 | 0.99 | 1.00 | 0.51† | 0.86 |

- **† cocoindex 의 낮은 정밀도는 거짓 답이 아니라 "줄 위치 오차"다.** 임베딩은 코드 '조각(청크)'의 줄 *범위*를 주므로, 호출지점 자체는 맞게 찾았는데 정확한 줄을 ±2 정도로 빗나간 제출이 있었다. 함정 포함·환각은 0건.
- **‡ codegraph 의 평균 정밀도(0.79)는 M3-L 을 뺀 3개 질문 평균**이다. M3-L 은 답을 0개 제출해(아무것도 못 찾음) 정밀도를 정의할 수 없어 분모에서 제외했다.
- **격리 준수**: A·B·C·F·H 완전 준수. D 는 `rg` 만 사용(sg 미사용). E(bare) 는 환경의 serena 정책 오염을 제거하고 텍스트+읽기로 측정(grep 을 Bash 경유 — 같은 ripgrep 엔진이라 능력 동등). 자세한 내용은 §8.

## 4. 질문별 결과 자세히

### M1 — 시그니처 변경 영향 (정답 33 = 프로덕션 12 + 테스트 21, 함정 2)

> **상황**: `formatJson(...)` 의 인자를 바꾸기로 했다. 이 함수를 호출하는 모든 곳을 빠짐없이 찾아야 같이 고친다.

| 도구 | prod | test | 재현율 | 오탐 / 누락 상세 |
|---|---|---|---|---|
| codegraph | 12/12 | 21/21 | **1.00** | 오탐 2: 이름만 같은 `presenter.formatJson`(Prettify:18, Uglify:20)을 포함 → 정밀도 0.94 |
| serena | **4/12** | 21/21 | 0.76 | 누락 8: 타입을 명시하지 않고 DI 로 받은 필드를 통한 호출 전부(`formatterService.*`, `service<T>()`) |
| zoekt | 12/12 | 21/21 | 1.00 | 함정 정확 배제. 정밀도 0.97(메서드 정의 1건 포함) |
| rg(+sg) | 12/12 | 21/21 | 1.00 | 함정 정확 배제. 정밀도 0.97 |
| bare | 11/12 | 21/21 | 0.97 | 누락 1: 함수 내부 자기호출(:215). **DI 수신자(:60)는 텍스트로 회수** |
| cocoindex | **9/12** | **13/21** | **0.67** | 누락: prod 3 + test 8(임베딩 상위 결과에 안 떠 누락). **함정 2 배제·환각 0** — 단지 전수 열거를 못 함 |
| zoekt+ctags | 12/12 | 21/21 | 1.00 | plain zoekt 와 동일. M1 은 ctags 없이 텍스트만으로도 충분 |

**핵심**: 이름으로 전수 검색만 잘하면 거의 만점인 쉬운 질문. 약점이 드러난 건 두 종류뿐 — **serena**(타입을 모르면 DI 호출을 못 봄)와 **cocoindex**(의미 검색이라 "33곳 전부"를 못 줌).

### M2 — 동작 전파 / transitive 폐포 (정답 8 함수)

> **상황**: `getOrCreate()` 의 동작을 바꾸면, 이걸 직접/간접으로 쓰는 기능이 다 영향받는다. 호출 사슬을 거슬러 올라가 "최종 영향 범위"를 구해야 한다.

| 도구 | 적중 | 재현율 | 누락 함수 |
|---|---|---|---|
| codegraph | 6/8 | 0.75 | `ConvertTypeDialogPresenter.<init>`(생성은 별도 엣지), `syncLanguage`(람다 콜백) |
| serena | 6/8 | 0.75 | 생성자 2개(`<init>`). DI 홉은 find_symbol 로 우회 회수 |
| zoekt | 5/8 | **0.625** | `bindView` + 생성자 2개 |
| rg(+sg) | 6/8 | 0.75 | `ConvertTypeDialogPresenter` 상단 1개 |
| bare | 5/8 | **0.625** | 생성자 2개 + `bindView` |
| cocoindex | 6/8 | 0.75 | 생성자 2개. 람다·DI홉은 반복 시맨틱 질의로 회수, 함정 4종 배제 |
| zoekt+ctags | 6/8 | 0.75 | 생성자 2개. **plain zoekt(0.625) 대비 개선** — `sym:` 심볼검색이 람다·DI홉 회수 |

**핵심**: 한 단계씩 거슬러 올라가야 하는 어려운 질문. **어느 도구도 8/8 을 못 했다** — 모두 사슬의 "마지막 한 홉"(생성자·람다 콜백)에서 샌다. 다만 plain zoekt 의 약점을 **ctags 가 0.625→0.75 로 보완**한 게 이 질문의 유일한 실측 개선이다.

### M3-W — 언어 간 경계 / ABI seam (정답 8 = Kotlin export 4 + Rust 정의 4, 함정 2)

> **상황**: Kotlin 이 Rust(WASM) 함수를 `instance.export("analyze_source")` 처럼 **문자열 이름**으로 부른다. 이 바인딩 지점과 대응하는 Rust `#[no_mangle] pub extern "C"` 정의를 양쪽 다 찾아야 한다.

| 도구 | 재현율 | 정밀도 | 판정 |
|---|---|---|---|
| codegraph | **0.50** | 0.44 | Rust 정의 4/4 는 찾음. 하지만 **Kotlin 쪽 문자열 바인딩 0/4 는 못 봄**(문자열은 심볼이 아니라 그래프에 없음) + 함정 2 포함 |
| serena | 1.00 | 1.00 | Rust=`find_symbol`, Kotlin=`search_for_pattern`(serena 에 내장된 텍스트 검색)으로 회수 |
| zoekt / rg / bare | 1.00 | 1.00 | 텍스트로 Kotlin·Rust 양쪽 정확 매칭 |
| cocoindex | **0.125** | 0.10 | 청크 줄 추정 편차 큼 — 필드/`.apply()` 정도만 회수 |
| zoekt+ctags | 1.00 | **0.67** | 정답 8 정확 + ExportFunction 필드 선언 4건을 더 보고해 정밀도만 하락 |

**핵심**: **"정밀 도구가 텍스트를 이긴다"는 통념이 뒤집힌 질문.** 순수 그래프인 codegraph 만 절반에서 무너진다(문자열로 부르는 건 그래프에 안 잡힘). serena 가 1.00 인 건 정밀해서가 아니라 **내장 텍스트 검색을 곁들였기 때문**이다.

### M3-L — enum 참조 (정답 29 = main 13 + test 16, enum 선언 줄은 분모 제외)

> **상황**: `SupportedLanguage.KOTLIN` 이라는 enum 상수를 실제로 쓰는 모든 곳(when 분기 포함)을 찾아야 한다. 함정은 정답이 아닌 문자열 `'kotlin'`·`KOTLIN_` 접두어·리소스 경로(텍스트로 'kotlin' 이 173줄/42파일에 깔려 있음).

| 도구 | 재현율 | 정밀도 | 판정 |
|---|---|---|---|
| codegraph | **0.00** | – | **그래프가 enum 상수 참조를 엣지로 색인하지 않아 한 개도 못 찾음**(스키마 한계) |
| serena | 1.00 | 1.00 | `find_referencing_symbols` 로 정밀 회수 |
| zoekt / rg / bare | 0.966 | 1.00 | `"SupportedLanguage.KOTLIN"` 정확 검색 → 점 없는 `KOTLIN`(:77) 한 곳만 누락 |
| cocoindex | 0.07 | 0.15 | 청크 줄 추정 편차로 ±1 매칭이 거의 실패 |
| zoekt+ctags | 0.966 | **0.76** | :77 누락 + 범위 밖 Rust `SupportedLanguage::Kotlin` 9건 과보고 |

**핵심**: 정밀 도구(serena)가 1등이지만, **텍스트 도구도 정규화된 토큰("SupportedLanguage.KOTLIN")으로 검색하면 0.966 까지 따라붙는다** — "정밀 우위"가 생각만큼 크지 않다. 반대로 codegraph 는 여기서 **완전히 0점**이다.

## 5. 도구별 총평

단일 지배 도구 없음. 질문 유형에 따라 강·약점이 갈린다.

| 도구 | 강점 | 약점 |
|------|------|------|
| **codegraph** | 한 언어 안의 전수 역호출·집계가 최강(M1·M2). 단순 질의 효율도 좋음 | 이름만 같으면 오탐(타입 맹점, M1 정밀도 0.94). **언어 경계·enum 참조에서 무너짐**(M3-W 0.50, M3-L 0.00) |
| **serena** | 타입 정밀(정밀도 0.97~1.00). 내장 텍스트 검색도 곁들여 M3 양쪽 강함 | **타입을 명시 안 한 DI 수신자 호출을 일관 누락**(M1 재현율 0.76) |
| **zoekt** | 단순 전수 조회를 최소 비용으로(M1 1콜·37s) | 다중홉 사슬의 말단 회수가 최저(M2 0.625) |
| **rg(+sg)** | 텍스트 컨텍스트로 높은 정확도(평균 재현율 1위 0.93) | 다중홉은 수동으로 거슬러 올라가야 함. 이번엔 sg(구조검색) 미사용 |
| **bare** | 추가 도구 없이도 단순 역호출엔 강(M1 0.97) | 깊은 transitive 는 최약체(M2 0.625). rg 와 같은 능력군이라 독립 우위 없음 |
| **cocoindex** | "의미가 비슷한 코드 찾기"엔 강. 함정·환각 0, serena 가 놓친 DI 호출도 회수 | **빠짐없이 다 찾기엔 부적합(평균 재현율 0.40 최저)**, 줄 위치 오차, 호출수·시간 최다 |
| **zoekt+ctags** | plain zoekt + ctags 심볼 메타(`sym:`). M2 람다·DI홉 회수 개선(0.625→0.75), M3 텍스트 강점 유지 | 생성자(`<init>`) 누락은 plain 과 동일. 심볼 필드 선언 과보고로 일부 정밀도 하락 |

## 6. 효율 — 호출 수·시간·토큰 (2026-06-01 전면 재측정, 동일 세션 14셀)

7암 × {M1,M2} = 14셀을 같은 세션에서 다시 측정해 각 서브에이전트 transcript 에서 토큰을 분리 집계했다(기존 표는 cocoindex·ctags 가 out/cacheR 로 나뉘지 않아 총량만 있었어서, 7암 통일 수치로 전면 교체했다).

> ⚠️ **N=1·고변동 — 절대값보다 상대 윤곽으로.** 이 재측정은 이 세션의 서브에이전트 하니스로 돌렸고 에이전트가 무겁게(검증·재질의 반복) 동작해 호출수·토큰이 큰 편이다(예: serena 572콜, codegraph 219콜). 케이스당 1회라 같은 암도 M1↔M2 변동이 크다 — 절대 효율 단정보다 **암 간 상대 비교**로 읽어야 한다. 정확도 스코어보드(§3)는 별도의 원 blind 측정이며 이 표와 무관하다.

- **out** = 출력 토큰, **cacheR** = 캐시읽기(매 턴 컨텍스트 재독, 입력 비용 대부분), **cacheCreate** = 캐시생성(첫 캐싱), **calls** = 도구 호출(하니스 집계, 재시도 포함), **time** = 소요(초).

**효율 관찰 (상대 비교):**
- **serena(B) M2 가 cacheR 34.5M 로 압도적 최대** — 다중홉 역참조를 546콜로 반복하며 컨텍스트를 매 턴 재독한 결과. "다중홉 폐포 추적이 가장 비싸다"는 정성 결론과 방향 일치.
- **다중홉(M2)이 단일 전수조회(M1)보다 호출·시간 폭증** — 전 암 공통(M1 17~64콜 → M2 145~546콜).
- **cocoindex(F)·ctags(H)는 cacheCreate 가 큼**(19.6M / 23.1M) — 큰 인덱스·컨텍스트의 첫 캐싱 비용. (이제 두 암도 out/cacheR 분리값이 존재 — n/a 해소.)
- **변동성 큼(N=1)**: 같은 암도 M1↔M2 토큰 차가 크다(예: rg out M1 3.2k → M2 56.8k). 절대 효율 단정은 피하고 상대 윤곽만 읽는다.

> **cocoindex 시간 주의**: cocoindex 가 느린 건 "검색이 느려서"가 아니라 `query.py` 가 **매 호출 임베딩 모델을 새로 로드**하는 콜드스타트 탓이다(실측 1회 7.5~9.7s 중 torch import + 모델 로드 ≈6.9s/90%, 실제 검색 0.4~0.8s). 모델 상주(서버) 방식이면 python 측은 ~20s 로 줄어든다. 인덱스 빌드 시간은 측정 밖이다(`query.py` 는 인덱싱을 호출하지 않음 — 코드 확인). → cocoindex 시간을 다른 암과 직접 비교 금지.

## 7. 핵심 발견 (왜 이런 결과가 나왔나)

### (1) bare(텍스트+읽기)는 rg 와 같은 능력군 — 독립 우위 없음
재현율 M1 0.97 / M2 0.625 로, 같은 계열인 rg(1.00 / 0.75)에 근소하게 못 미친다. 단순 역호출엔 충분하나 깊은 transitive 에선 최약체. "기본 도구만으로 충분하다"는 일반화는 성립하지 않는다(질문 규모·유형에 따라 달라짐).

### (2) codegraph 의 타입 맹점 (M1 정밀도 0.94)
이름만으로 매칭해 동명 `JsoninjaPanelPresenter.formatJson` 호출 2건을 오탐했다. 단 재현율은 1.00(테스트 꼬리 21건까지 전부 회수) — **전수성은 강점, 타입 구분은 약점**.

### (3) serena 의 DI 약점은 형태와 무관하게 일관됨 — M1·M2 차이는 "도구 한계"가 아니라 "에이전트 행동" (raw 출력 직접 검증)
처음엔 "M1 에선 약점이 나오고 M2 에선 안 나오니 DI 형태에 따라 다른가" 싶었으나, **serena 의 raw 출력을 직접 실행해 보니 그 해석은 틀렸다.**
- **도구 한계는 M1·M2 동일**: `find_referencing_symbols` 를 직접 돌리면 M1 은 프로덕션 12개 중 4개만, M2 는 프로덕션 DI 홉을 둘 다 반환하지 못한다.
- **진짜 원인**: 누락된 호출의 수신자는 전부 **타입을 명시하지 않고 DI 로 받은 val** 이다(`service<T>()` reified, `getService(...::class.java)` 둘 다 동일하게 누락). 적중한 건 수신자에 **명시적 타입**이 붙은 경우뿐. → IntelliJ Platform SDK 가 분석 classpath 밖이라 **LSP 가 DI 반환 타입을 추론 못 하는 것**이 공통 원인. 타입 주석을 달면 해소된다.
- **M1(0.76) vs M2(0.75) 차이 = 에이전트 행동**: M1 에이전트는 도구의 누락을 그대로 보고했고, M2 에이전트는 `find_symbol` 로 파이프라인을 탐색해 누락된 홉을 **의미론적으로 우회 복원**했다.
- **교훈**: "도구의 원시 출력 ≠ 에이전트가 그 도구로 도달한 답". 도구의 맹점은 일관되며, 유능한 에이전트가 부분적으로 우회할 뿐이다.

### (4) 다중홉 사슬의 말단(생성자·람다 콜백)은 어떤 단일 도구도 지배 못 함 (M2)
정답 8함수 중 깊이 4~5 말단에서 누락 양상이 도구마다 달랐다:
- **codegraph**: 생성(`<init>`)은 호출과 다른 엣지(`instantiates`)라 역호출 추적에서 빠지고, 람다 콜백(`{ syncLanguage(...) }`)은 정적 호출 엣지로 안 잡힘.
- **serena**: 람다·DI홉은 우회했으나 생성자 2개 누락.
- **zoekt·bare(최저 0.625)**: 텍스트 매칭이라 람다/생성자 블록의 호출 귀속이 가장 약함.
- **rg**: 한 생성자만 누락.
→ 사슬의 "마지막 한 홉"에서 모두 서로 다른 이유로 샌다.

### (5) (M3-W) 언어 간 경계 — 정밀 도구가 텍스트에 역전당함
Kotlin 이 Rust export 를 **문자열 이름**으로 바인딩하는 지점에서, 텍스트군(zoekt/rg/bare/ctags)·serena 는 1.00 인데 순수 그래프 **codegraph 만 0.50**(문자열은 심볼이 아니라 미색인; Rust 정의 4만 회수). serena 가 만점인 건 내장 텍스트 검색(`search_for_pattern`)을 겸비했기 때문 → **"정밀 도구 전멸"은 codegraph 한정**. M1·M2 의 "정밀 > 텍스트" 서열이 여기서 뒤집힌다.

### (6) (M3-L) enum 참조 — codegraph 0.00, 텍스트도 정밀에 근접
`SupportedLanguage.KOTLIN` 참조(정답 29)에서 serena 는 1.00 이지만, **codegraph 는 0.00**(그래프가 enum 상수 참조를 엣지로 색인하지 않음). 텍스트군도 정규화 토큰 검색으로 0.966/1.00 → **정밀 우위는 약했다**(정확한 토큰을 쓰면 함정 173줄도 회피). 만약 소문자 `'kotlin'` 같은 느슨한 검색을 강제했다면 텍스트군의 정밀도 격차가 더 컸을 것이다.

### (7) cocoindex(의미 검색): 전수·정확 회수엔 부적합하나 무너지진 않음
*정확 매칭형* 도구들과 능력 축이 다르다.
- **M1 에서 최저 재현율 0.67** — 임베딩 상위 결과는 "비슷한 청크"를 줄 뿐 "X 를 호출하는 33곳 전부"를 못 준다. **전수 회수에 구조적으로 부적합함을 데이터로 확인.**
- **그러나 함정·환각 0, DI 회수** — 청크를 직접 읽으므로 동명 함정을 정확히 배제하고, serena 가 일관 누락한 DI 호출도 회수했다(임베딩은 타입 추론에 의존 안 함). 정밀도 하락은 거짓 답이 아니라 **줄 위치 추정 오차**.
- **M2 에서 0.75(최상위 동률)** — 의외다. 에이전트가 각 홉을 시맨틱 질의로 찾고 청크로 검증하는 **수동 시맨틱 BFS**를 30회 수행해 보완. 대가는 최다 호출수·최장 시간.
- **요약**: "이 기능과 의미적으로 관련된 코드 찾기"엔 강하지만(설계 목적), 전수·정확 회수엔 약하다 — **"만능 도구 없음"을 반대 방향에서 보강**한다.

### (8) 비단조성의 실체 = "정밀 vs 텍스트"가 아니라 "무엇을 색인했나"
4개 질문을 가로지르면 단일 지배 도구가 없다(평균 재현율: rg·ctags 0.93, zoekt 0.90, bare 0.89, serena 0.88, **codegraph 0.56, cocoindex 0.40**). 갈림의 축은 타입 정밀도가 아니라 **백엔드가 색인 가능한 대상**(문자열 리터럴 / enum 참조 / 언어 간 링크 / 줄 정밀도)이다. codegraph 는 한 언어 전수성은 최강이나 polyglot 경계에서 무너지고, ctags 는 plain zoekt 의 M2 약점만 보완한다.

## 8. 신뢰성 — 채점을 어떻게 공정하게 했나

- **정답(gold)은 소스에서 독립 산출** — 측정 대상 7개 도구 중 어느 것도 정답 산출에 쓰지 않았다(순환논리 차단). 홉별 도달 경로를 `dataset.yaml` 에 감사 가능하게 기록.
- **정답 비공개(blind) 채점** — 에이전트에 정답을 안 줬다. 줄 ±1 정규화로 도구 간 줄 번호 기준 차이를 흡수.
- **provenance 고정** — git `v1.12.1`(`1daf879`). M1/M2 codegraph DB sha256 `54e743b6…`, M3 인덱스는 Rust(`tree-sitter-wasm/`) 포함으로 재구축(`PROVENANCE-m3.txt`).
- **범위 정렬** — 정답에 테스트 호출지점이 포함되므로 zoekt 등을 src(main+test)로 재인덱싱해 공정성 확보. M3-W 가 Rust 정의를 포함하므로 전 암을 Rust 포함으로 통일(A안).

### 격리 감사 (각 도구가 자기 허용 도구만 썼는지 사후 확인)
- **A codegraph**: `Bash(sqlite3)` 만 → 준수.
- **B serena**: `mcp__serena__*` 만 → 준수.
- **C zoekt / H zoekt+ctags**: `Bash(zoekt)` 만 → 준수.
- **D rg(+sg)**: `Bash(rg)`(+M2 에서 `Read`) → 텍스트 계열 준수. 단 **sg 미사용**.
- **E bare**: serena 정책 오염 제거됨(최우선 anti-hook 지시 효과). 단 grep 을 내장 `Grep` 대신 `Bash grep` 경유 — 같은 ripgrep 엔진이라 능력 동등.
- **F cocoindex**: 전부 `python query.py` (소스 직접 Read/grep/sqlite3 0건) → 완전 준수.
- **M3 신규 16셀**: 각 에이전트가 허용 도구와 일치(소스 직접 판독 0건 확인). 단 현 세션 서브에이전트 transcript 경로 문제로 **전수 grep 감사는 미완** — "0 위반"은 아직 정식 확인 전(보수적 표기).

## 9. 한계

- **각 질문 1회 실행(N=1)** — 샘플링 분산은 측정하지 않음.
- **cocoindex(F)**: 임베딩 모델·청크 크기(1000B)·top-k 가 결과를 좌우하는 새 변수(본 실행 = all-MiniLM-L6-v2 고정). 코드 특화 모델·청크 조정 시 M1 재현율이 달라질 수 있다. 출력이 청크라 (file:line) 채점에 줄 추정이 끼며, 정밀도 하락분은 거짓 답이 아니라 위치 추정 오차다.
- **D 암 sg 미사용**(rg-only) — 구조 검색(ast-grep)의 변별력은 여전히 미측정.
- **A codegraph 의 M3 인덱스는 신규 도구버전(@colbymchenry 0.9.7)으로 재빌드** — 기존 M1/M2 의 codegraph(sha `54e743b6`)와 스키마/버전이 다를 수 있다(M1/M2 점수는 기존 재사용, M3 만 신규).
- **채점 컨벤션**: M2 정답의 `bindView` 는 런타임에 직접 호출하는 게 아니라 콜백을 등록하는 함수다(렉시컬 포함 기준으로 폐포에 넣음). 전 암을 동일 기준으로 채점했으므로 상대 비교는 유효하다.
- **토큰·시간은 서브에이전트 usage 기준 추정치.**
